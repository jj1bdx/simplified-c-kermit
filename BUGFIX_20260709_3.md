# C-Kermit 10.0 Beta.12 (simplified fork) — revert of the SPACMD workaround (BUGFIX_20260709)

**Subject:** commit `6d3f1a4` reverts commit `7f77234` — the `ckufio.c` `SPACMD` change from
`BUGFIX_20260709.md` — because the hang originally reported against the macOS server (`daimac`)
is now attributed to the `spack()` packet-framing defect that was found and fixed the same day in
commit `c6e9bbc` (`BUGFIX_20260709_2.md`). This document records the reasoning behind the revert.

- **Branch:** `fix-20260708` (HEAD = `c6e9bbc` + revert `6d3f1a4` + this doc)
- **Date:** 2026-07-09
- **Companion docs:** `BUGFIX_20260709.md` (the reverted change; now carries a notice pointing
  here), `BUGFIX_20260709_2.md` (the actual fix).

---

## TL;DR

1. **What was reverted:** the `SPACMD` guard in `ckufio.c` is back to the original upstream
   `#ifdef BSD4` (dead on every live target), so the no-argument `REMOTE SPACE` server command on
   macOS/BSD once again runs a plain **all-volumes `df`** — the original upstream behavior — with
   a source comment recording both commits. Linux is unaffected in both directions (`SPACMD` is
   `"df "` under either guard).
2. **Why:** the daimac hang that motivated `7f77234` is reattributed to the `spack()`
   short/long-packet boundary defect (`BUGFIX_20260709_2.md`): the sender framed any chunk with
   `len + bctl + 2` ∈ {95, 96} as a short packet with an illegal LEN byte (DEL / 0x80), which the
   receiver's framer discards without replying — a silent multi-minute stall. That defect was
   *observed in packet logs* (mirror direction, gluonite as server) and is fixed at the root;
   the blocking-volume mechanism of `BUGFIX_20260709.md`, by contrast, was never observed to
   fire on daimac.
3. **The workaround was therefore unnecessary** — and it changed user-visible macOS behavior
   (single-filesystem `pwd ; df .` instead of the all-volumes listing that `REMOTE SPACE` is
   meant to produce) to guard against a hazard that remained hypothetical.
4. **Post-revert verification:** both builds warning-clean; live end-to-end test against the
   rebuilt daimac server — no-arg `remote space` returns the full multi-volume `df` (including
   the `devfs` line that only the all-volumes listing contains) immediately, with `remote space
   /tmp` and `remote directory` regression checks passing.

## 1. Why the reattribution is the better explanation

The original report (2026-07-08/09): no-arg `REMOTE SPACE` from a Linux client against the
daimac (macOS) server "hangs up" the session; the with-argument form does not. Two candidate
mechanisms emerged, one per investigation:

| Observation | Blocking-volume theory (`BUGFIX_20260709.md`) | `spack()` framing defect (`BUGFIX_20260709_2.md`) |
|---|---|---|
| Session hangs silently, no error, no output | Fits (unbounded wait, forever) | Fits (33 silent retransmissions, `E` packet only after ~4½ min) |
| Intermittent: hung for the reporter, passed in every scripted repro | Requires a sleeping/wedged volume — **none was ever observed**; `df` completed in ~3 ms in all tests | Trigger is pure data size: the transfer's *final chunk* must land on exactly 90/91 data bytes (block check 3). `df` output drifts by a byte or two as block counts change, so the same command can hang one day and pass the next |
| Mechanism ever captured in evidence? | Only by **simulation** (`remote host sleep …` — proves the machinery *can* wait forever, not that it did) | **Yes** — the mirror-direction report (daimac client → gluonite server, same command, same symptom) reproduced deterministically and the server packet log shows the malformed 0x80-LEN packet |
| Why only the no-argument form? | Single-filesystem `df <dir>` never stats other volumes | The specific output sizes at hand: the no-arg listing's tail happened to land on the 2-byte window; `df /tmp`'s did not |

The deciding facts: the `spack()` defect is a real, logged, deterministic-per-size bug whose
symptom exactly matches the report, while the blocking volume is a condition nobody ever caught
in the act; and the mirror-direction incident proved the same command/symptom pair arises purely
from output size, with direction and OS incidental.

**Honest caveat:** the original daimac-server hang was never captured in a packet log, so the
reattribution cannot be proven retroactively — daimac's `df` output size on 2026-07-08 is
unknowable now. It is, however, the only mechanism found that is consistent with every
observation, and it is fixed at the root by `c6e9bbc`. If the hang ever recurs on a current
build, the blocking-volume theory becomes interesting again (see §4).

## 2. The revert

`6d3f1a4` is a *manual* revert (not `git revert`):

```c
/*
  This was once forced to use "df ." in commit
  7f77234007945d744ccf135658a2e9cf150f035b, but is manually reverted
  because the true cause of the bug was NOT by the command.
  See BUGFIX_20260709.md for the past reasoning.
  The actual cause is described in
  commit c6e9bbc606937514fed60dab11daeee361565812
  See BUGFIX_20260709_2.md for the past reasoning.
*/
#ifdef BSD4
char *SPACMD = "pwd ; df ."; /* Space in current directory */
#else
char *SPACMD = "df ";
#endif /* BSD4 */
```

- `BSD4` is defined by no live target (macOS implies `BSD44`, not `BSD4` — see
  `BUGFIX_20260709.md` §TL;DR-3), so all platforms use `SPACMD = "df "`: the pristine upstream
  behavior, dead branch and all.
- The commit also rewraps the two long-standing whitespace-unstable comment hunks in `ckufio.c`
  (~lines 5135 and 6027), so the file is now **fully clang-format-stable at HEAD** for the first
  time.
- User-visible effect (macOS/BSD only): no-arg `REMOTE SPACE` reports **all mounted volumes**
  again, which is what the command is designed to do.

## 3. Verification

- **gluonite:** `make linux` after the revert — exit 0, warning-clean (behaviorally a no-op for
  Linux). `clang-format ckufio.c | diff` — empty (whole file stable).
- **daimac:** already at `6d3f1a4` with `wermit` rebuilt from the reverted source before the
  commit was made — confirmed by `strings wermit`: the binary no longer contains `"pwd ; df ."`.
- **End-to-end over real ssh** (gluonite client → daimac server, post-revert binaries both
  sides): no-arg `remote space` returns the full multi-volume `df` immediately — the response
  contains the `devfs` line, present only in the all-volumes listing, proving the revert is live
  end-to-end; `remote space /tmp` and `remote directory` regression checks pass; `finish` clean.
  This also exercises the `c6e9bbc` framing fix with today's (larger) all-volumes output.

## 4. Residual notes

1. **The blocking-volume hazard is back to its natural scope.** If a sleeping/wedged external or
   network volume ever does make `df` block, the no-arg `REMOTE SPACE` will wait unboundedly —
   but that was never specific to `SPACMD`: *every* `syscmd()`-based server command (`REMOTE
   HOST/WHO/DIRECTORY/TYPE/PRINT`, and `SPACE <dir>` against a stuck filesystem) has the same
   silent 1 Hz empty-packet wait (`BUGFIX_20260709.md` §2 and §5.1). Reverting one command's
   workaround does not change the general exposure; a real fix would be a cumulative no-output
   cap in the shared `sdata()`/`zinfill()` polling loop, deliberately not attempted.
2. **Triage hint for any future "REMOTE xxx hangs" report:** get a `log packets` capture first.
   Identical retransmissions with timeouts on both sides and no NAKs = malformed packet
   (framing/encoding bug); 1 Hz *empty* data packets, each ACKed = blocked server-side shell
   command. The two mechanisms look identical on screen and completely different in the log.
3. The dead `#ifdef BSD4` branches (`SPACMD`, and its sibling `WHOCMD`) are now deliberately
   preserved upstream-verbatim dead code; do not "fix" them to `BSD44` again without new
   evidence that the all-volumes `df` itself is a problem in practice.
