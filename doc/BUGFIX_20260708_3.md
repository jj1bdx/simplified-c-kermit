## TL;DR — honest status

**No `fix`-vs-`main` difference exists for no-argument `REMOTE SPACE` — verified over a real ssh
connection.** Tested against an actual remote host (`gluonite`, Linux x86_64) with the real `ssh`
client as the transport (`set host ssh -t gluonite <wermit> -x`): no-arg `REMOTE SPACE` **succeeds**
in the `fix-20260708` build, identically to `main` (rc=0, full `df`, clean `Z`+`B`), for both a fix
server and a main server.

**The one real over-ssh failure mode found is NOT a fix regression:** if ssh is invoked *without a
pseudoterminal* (no `-t`), the remote `wermit -x` server prints **"/dev/tty is not a terminal
device"** (`ckutio.c:1978`, in `ttopen()`) and exits — so the client gets no reply. This happens
**identically in `main` and `fix`** (confirmed by running both remotely), in code this branch never
touched. It is a pre-existing requirement that `kermit -x` have a terminal; the fix is to allocate a
pty (`ssh -t`, which interactive ssh does automatically), not a source change.

Because there is no reproducible `fix`-specific defect, I will **not** invent a speculative code
change — that would risk breaking a working protocol path. This document records the full
investigation (now including real ssh), the two real fixes already in place, and — if the user's
setup still fails after a clean rebuild with a pty — a precise diagnostic procedure.

## 1. What was tested (all PASS)

`REMOTE SPACE` runs a shell command on the server (`SPACMD="df "` on Linux; `"pwd ; df ."` on
BSD/macOS) and streams the output back — the same machinery as `REMOTE DIRECTORY`. Tested with the
`socat`-PTY loopback and with ssh's real transport:

| Transport | Args | server / client | Result |
|---|---|---|---|
| serial (`set line`) | none | fix/fix, fix/main, main/fix | **PASS** (full `df`) |
| **NET_PTY** (`set network type pty` + `set host … -x`; what ssh uses) | none | fix/fix, main/fix | **PASS** (`rc=0`, full `df`, clean `Z`+`B`) |
| NET_PTY + streaming + no control-prefixing + window 30 + 9024-byte packets | none | fix, main | **PASS** |
| NET_PTY, macOS-style `SPACMD="pwd ; df ."` | none | fix | **PASS** (`rc=0`, clean) |
| serial / NET_PTY | `/home`, `.`, `/tmp`, `/nonexistent` | fix/fix | **PASS** |
| **real ssh** to `gluonite`, **with pty** (`set host ssh -t gluonite <wermit> -x`) | none | fix/fix, fix/main | **PASS** (`rc=0`, full `df`) |
| **real ssh** to `gluonite`, **no pty** (no `-t`) | none | fix *and* main | **FAIL — identical:** server exits with `/dev/tty is not a terminal device`. Pre-existing (`ttopen()`), **not** a fix regression. Use `ssh -t`. |

> ssh in C-Kermit runs the remote end over a pseudoterminal (`set host /pty ssh host "kermit -x"`),
> i.e. **`NET_PTY`** — which is exactly what "NET_PTY" above spawns. Streaming did **not** engage over
> `NET_PTY` (`streamok=0` in the debug log), consistent with real ssh, so streaming is not the
> trigger. `alrm_execute()`/`cc_alrm_execute()` (the V-42 signal-order change) are used only for
> dial/script/network-*connect* timeouts, **not** the packet-transfer path.

## 2. Why the earlier reports were real — and are already fixed

- **`REMOTE DIRECTORY`** and **`REMOTE SPACE`** were both broken by the V-3 `xunchar()`
  multiple-evaluation regression (`xunchar(recpkt[i++])` advancing the index 2–3×). Verified: both
  fail on `1356430` (pre-fix) and both work on `7ec09ca` (post-fix). See `BUGFIX_20260708.md`.
- A real **makefile dependency gap** (`ckclib`, `ckucmd`, `ckufio`, `ckutio` omitted `ckcker.h`) was
  found and fixed so incremental builds after a `ckcker.h` edit recompile every dependent object.
  See `BUGFIX_20260708_2.md`.

Both fixes are in the tree. With them, no-arg `REMOTE SPACE` works in all the configurations above.

---

## 3. Most likely resolution: a clean rebuild

The single most probable reason an "updated" `fix-20260708` tree would still fail is a **stale
object build**. `make linux` is incremental and the tree's dependency lists were incomplete (now
fixed). Do:

```sh
git checkout fix-20260708
make clean          # discard all stale .o
make wart && make ckcpro.c   # regenerate the wart output (safe to always do)
make linux
```

A from-scratch build guarantees every object reflects the `xunchar()` fix. In testing, a clean build
makes both `REMOTE SPACE` and `REMOTE DIRECTORY` work over serial and over ssh's `NET_PTY` transport.

## 5. Candidate causes to check against those logs

None of these reproduced here; the logs will confirm or eliminate each:

1. **Stale/partial build** (section 3) — most likely. The debug banner shows the build; confirm it is
   a clean build at `7ec09ca` or later.
2. **A clean server-side refusal misread as a comms failure** — `df`/push disabled on the server →
   `errpkt("Can't check space")`; `DISABLE SPACE` → `"REMOTE SPACE disabled"`. These are expected
   error replies, not protocol failures.
3. **Platform-specific (macOS/BSD)** — `SPACMD="pwd ; df ."` and the `SVR4/BSD44/MACOSX` code paths
   are untestable in this Linux sandbox. The `xunchar()` fix is platform-independent (a header macro),
   so it should apply equally, but a macOS-only issue can only be confirmed on macOS.
4. **A specific byte/size in the server's real `df` output** — a busy server with many mounts / long
   device paths produces much larger output than the 14-line sample here; the packet log will show if
   a particular packet crunches.
