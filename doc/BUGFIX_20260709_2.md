# C-Kermit 10.0 Beta.12 (simplified fork) — spack() emits unanswerable packet at the short/long-packet boundary

**Subject:** `REMOTE SPACE` *without arguments* against a Linux server (`gluonite`), from a macOS
client (`daimac`), hangs the session — the mirror direction of `BUGFIX_20260709.md`, but an
entirely different (and more fundamental) bug: a **packet-framing defect in `spack()`** that can
stall *any* transfer whose final chunk lands on an unlucky size.

- **Branch:** `fix-20260708` (HEAD = `7f77234` + this fix)
- **Date:** 2026-07-09
- **Companion docs:** `BUGFIX_20260709.md` (macOS-server `df` hang; SPACMD), `BUGFIX_20260708.md`
  (`xunchar()` regression).

---

## TL;DR

1. **Symptom:** client sends no-arg `REMOTE SPACE`; the server's `df` output goes out as
   extended-length D packets; the **final short chunk is framed as a normal packet whose LEN byte
   is 0x80** — not a legal printable ASCII field. The client's framer discards it without ever
   replying (not even a NAK), the server retransmits the identical bytes 33 times at its timeout
   interval, and after ~4½ minutes gives up with `E "Too many retries"`. To the user the session
   is simply hung.
2. **Root cause:** `spack()` (`ckcfn2.c`) switched to extended-length format only when
   `len + bctl + 2 > 96`. But the short-packet LEN field is `tochar(len+bctl+2)`, legal only for
   values **0..94** (`tochar(94)` = `~` = 126, top of the printable range). Values 95 and 96 took
   the short path and encoded as DEL (127) and 0x80 — both of which the receiving side's own
   framer (`ttinl()`, `ckutio.c`) has always rejected as "Garbage/Rubout in length field".
3. **Trigger window:** the failure needs the final chunk of a response/file to hit
   `len + bctl + 2` ∈ {95, 96} — with block check 3 (this fork's default negotiation), data
   lengths of exactly 90 or 91. Pure data-size luck: gluonite's `df` output hits it, small outputs
   and `remote space <dir>` don't. **Direction, OS, and command are incidental** — any transfer
   (file, `REMOTE DIRECTORY`, etc.) whose tail lands on the window stalls the same way.
4. **Upstream bug**, verbatim in pristine C-Kermit 10.0 Beta.12 (initial commit `5a8fc5a`),
   attached to an fdc revision dated 13 September 2022. Not a simplification or security-fix
   regression.
5. **Fix:** change the threshold to `> 94` so sizes 95/96 use extended-length framing (legal,
   already negotiated in any session that can reach this code — the branch is guarded by
   `lpcapu > 0`).

## 1. Reproduction and evidence

Scripted the reported flow with `expect` over real ssh (client `./wermit` on daimac → `ssh
gluonite` → `./wermit`, `server` → escape back → `remote space`): **hung on the first try**, and
deterministically thereafter. Server-side packet log (gluonite):

```
s-03-00-^A #D"Y"Filesystem~% 1K-blocks ...      <- extended-length D packets, ACKed fine
s-04-00-^A $D"Y#7700 ...
s-05-00-^A %D"Y$320 202503400 ...
s-06-00-^A<0x80>&Dun/credentials/getty@tty1...  <- final chunk: SHORT packet, LEN byte 0x80
r-xx-01-<timeout>                               <- no reply, ever
S-06-01-^A<0x80>&D...                           <- identical retransmission
    ... 33 identical retransmissions ...
s-07-33-^A5'EToo many retries#57
```

Hexdump of the bad header: `01 80 26 44` = SOH, **0x80**, `&` (seq 6), `D`. The data field is 91
bytes; 91 + 3 (block check) + 2 (seq+type) = 96; `tochar(96)` = 128 = 0x80. The client-side log
(daimac) shows the mirror image: ACKs for 03–05, then 33 receive timeouts — the client never
even perceives a packet arriving.

**Why the client stays completely silent** (c-expert analysis, all receive-side):
`ttinl()` (`ckutio.c`) validates the LEN byte as `(dest[1] & 0x7f) < 32` → 0x80 masks to 0 →
"garbage" → return -1 *before packet parsing*; `rpack()` maps -1 to a soft timeout `'T'`; and
`input()`'s timeout handler skips sending a NAK whenever more input bytes are already pending
(`ttchk() > 0`) — which is almost always true while the server keeps retransmitting. Result: a
sender bug that a receiver, by design, cannot complain about.

## 2. The fix

`ckcfn2.c`, `spack()` long-packet decision (guarded by `lpcapu > 0`):

```c
-    longpkt = (len + bctl + 2) > 96;
+    longpkt = (len + bctl + 2) > 94;
```

with a comment recording the derivation (LEN is one printable character; `tochar()` is legal only
for 0..94; the receiver's `ttinl()` discards LEN > 94 without replying). Sizes 95/96 now go out
as small extended-length packets, which the receive path handles identically to the large ones
(`rpack()`/`ttinl()` extended-length framing is magnitude-agnostic).

## 3. Verification

- **Sensitivity check first (unfixed build):** a boundary sweep — `remote host head -c N
  <incompressible 248-char alphanumeric cycle file>` for N = 86..94, so the whole response is one
  D packet with `len+bctl+2 = N+5` — produced LEN bytes `{ | } ~` for N=86..89 (all pass) and
  **hung at exactly N=90** (LEN = DEL). Two earlier sweep designs were defeated by the protocol
  itself and are worth remembering: runs of a repeated character are collapsed by Kermit
  repeat-count compression (`~va` = 86×`a`), and a `|` in a REMOTE command is *client-side*
  redirection stripped by `remtxt()` before the command is sent.
- **Post-fix (make linux, warning-clean):** the same sweep passes for all N = 86..94, with
  N ≥ 90 framed as extended-length packets; the original df repro passes — no-arg `remote space`
  from daimac returns the full 14-line `df` immediately, then `remote space /tmp` and `finish`
  clean.
- **macOS side:** fix pulled to daimac, `make macos` warning-clean, and the daimac-as-server
  direction re-verified (no-arg `remote space`, `remote space /tmp`, `remote directory`).
- `clang-format ckcfn2.c` — edited region format-stable (the one whitespace-unstable comment hunk
  at ~line 1762 pre-exists at HEAD).

## 4. Out of scope / residual notes

1. **`lpcapu == 0` residual gap:** when long packets are *not* negotiated there is no extended
   format to escape to, and `sdata()`'s flat clamp (`if (spsiz <= 94 && spsiz > 90) spsiz = 90;`,
   `ckcfns.c`) is not block-check-aware: with block check 3, data can still reach 90 →
   `len+bctl+2 = 95` → DEL. Unreachable between two instances of this fork (long packets are
   always negotiated); reachable only against an old/limited peer negotiating CRC-16 without long
   packets. The dead function `maxdata()` (`ckcfns.c`, zero callers) already contains the correct
   bctl-aware formula if this is ever worth closing.
2. **Receive-side silence:** the `ttinl()` → `rpack()` → `input()` chain above converts any
   corrupt-LEN packet into an infinite mutual wait instead of a NAK; also `ttinl()`'s two
   garbage-LEN early returns skip `ttimoff()`, leaving the alarm timer armed (latent, masked by
   re-arming on the next call). Left untouched: fixing the sender removes the only known trigger.
3. The 33-retransmission cap that finally produced the `E` packet is `resend()`'s server-side
   retry counter — without it both sides would wait forever.
