# C-Kermit 10.0 Beta.12 (simplified fork) — Regression Analysis & Fix

**Subject:** `REMOTE DIRECTORY` (and, in fact, *all* Kermit-protocol operations) fail between the
`fix-20260708` build and a `main` build — and even between two `fix-20260708` builds.

- **Branch:** `fix-20260708`
- **Date:** 2026-07-08
- **Regressing change:** **V-3** (the `xunchar()` clamp in `ckcker.h`), introduced in Phase 1 commit `7cf795c`.
- **Companion docs:** `VULNERABILITIES-20260707.md`, `PROPOSED-FIXES-20260708.md`, `FIXES-20260708.md`.

---

## 1. Symptom

`remote directory` against a Kermit server returns **"Protocol error"** and no listing. The
handshake never completes: the client's first `I` (init) packet is repeatedly **NAK'd** by the
server until "Too many retries".

This is **not** specific to `REMOTE DIRECTORY`. `REMOTE DIRECTORY` was simply the command used to
notice it. The failure is in the packet **receive** path, so it breaks essentially every
Kermit-protocol exchange (SEND, GET, all REMOTE commands). It also breaks **`fix ↔ fix`**, not just
`fix ↔ main` — this is a local packet-parsing corruption, not a wire-format disagreement between
versions.

---

## 2. Reproduction

Two `wermit` instances were bridged over a `socat` PTY pair, one running `server`, the other issuing
`remote directory`, with `log packets` / `log debug` on both sides. Results:

| Server | Client | Result |
|---|---|---|
| `main` | `main` | **PASS** — full directory listing |
| `fix`  | `fix`  | **FAIL** — server NAKs the `I` packet; "Protocol error" |
| `fix`  | `main` | **FAIL** |
| `main` | `fix`  | **FAIL** |

The client's `I` packet is **byte-identical** on the wire in the passing and failing runs
(`^A9 I~& @-#Y3~^>J)0___F"U1@,`). Only the server's *handling* of it differs.

**Server-side packet log (fix, failing):**
```
r-...-^A9 I~& @-#Y3~^>J)0___F"U1@,          <- receives the I packet
r-...-<crunched:chk1>                        <- block-check (checksum) MISMATCH
s-...-^A# N3                                 <- sends NAK
```
**Server-side debug (fix, failing):**
```
rpack bctl=1 / rpack chklen=1
block check (1)=0        <- xunchar(received check byte) came out 0
should be (1)=11         <- computed checksum was 11  ->  mismatch  ->  NAK
```

**Bisect:** rebuilding the `fix` tree with *only* the `xunchar()` macro reverted to its original form
makes every combination **PASS** again. That isolates the regression to **V-3**.

---

## 3. Root cause — a multiple-evaluation macro over side-effecting arguments

V-3 replaced the original one-line macro:

```c
#define xunchar(ch) (((ch) - SP) & 0xFF)          /* evaluates (ch) ONCE */
```

with a clamping ternary:

```c
#define xunchar(ch) (((ch) < SP || (ch) > (SP + 94)) ? 0 : (((ch) - SP) & 0xFF))
```

The clamping form **evaluates its argument up to three times** — twice in the `||` condition and
once in the result. For an *in-range* byte (the normal case) all three evaluations happen.

That is fatal because **nearly every caller passes a side-effecting expression.** There are 12 such
call sites; the ones on the hot receive path are:

| Site | Code | Effect under multi-eval |
|---|---|---|
| `ckcfn2.c:2672` | `j = xunchar(recpkt[i++]);` — **packet length** | `i` advances **3×**; length read from the wrong byte |
| `ckcfn2.c:2709` | `rsn = xunchar(recpkt[i++]);` — **sequence number** | `i` advances 3× |
| `ckcfn2.c:2449` | `len = xunchar(*s++);` | `s` advances 3× |
| `ckcfns.c:1278` | `rpt = xunchar(*xdbuf++ & 0xFF);` — decode repeat count | `xdbuf` advances 3× |
| `ckcfns.c:560`  | `rpt = xunchar(*xdbuf++ & 0xFF);` (bdecode) | `xdbuf` advances 3× |
| `ckcfn3.c:1274`, `1385`; `ckcfns.c:6520`; `ckcpro.w:2622`, `2629`, `3465`, `3466` | `xunchar(*p++)` etc. | pointer advances 2–3× |

Concretely, in `rpack()` the length decode

```c
if ((j = xunchar(recpkt[i++])) == 0) { ...
```

reads `recpkt[i]`, `recpkt[i+1]`, `recpkt[i+2]` while incrementing `i` three times, so `j` is the
`xunchar` of the **TYPE** byte, not the **LEN** byte, and `i` is left three positions too far. Every
subsequent field — sequence, data span, block-check position — is misaligned, so the receiver
computes a checksum over the wrong bytes and reads the check digit from the wrong offset. The check
fails → NAK → the transfer never starts.

**Why the review missed it:** `PROPOSED-FIXES-20260708.md` explicitly flagged the sites that *combine*
two `xunchar()` results arithmetically (CRC assembly, extended length) and confirmed none relied on
the mod-256 wrap — but it did **not** consider that turning a single-evaluation macro into a
multiple-evaluation one is unsafe for the many `xunchar(*p++)` / `xunchar(recpkt[i++])` call sites.
The byte-identical-rebuild gate could not catch it either, since V-3 was an intentional behavior
change and no loopback protocol transfer was run.

---

## 4. The fix

Two parts. The security intent of V-3 is preserved; the protocol breakage is removed.

### 4a. Restore single-evaluation `xunchar()` (`ckcker.h`)

```c
/* before (V-3, broken): multiple-evaluation clamp */
#define xunchar(ch) (((ch) < SP || (ch) > (SP + 94)) ? 0 : (((ch) - SP) & 0xFF))

/* after: original single-evaluation form (identical to main) */
#define xunchar(ch) (((ch) - SP) & 0xFF)
```

A value-returning expression macro cannot both clamp and evaluate its argument once without a
non-portable statement-expression (`({ ... })`), which this codebase avoids. So the macro is
restored to the proven single-evaluation form, and the actual vulnerability is closed at its point of
use instead (4b).

### 4b. Clamp the pad count at its point of use (`ckcfns.c`, `spar()`)

The one place V-3 was really guarding is the S/I-packet **pad count**: `npad = xunchar(s[3])` feeds an
unbounded `for (i = 0; i < npad; i++) padbuf[i] = ...` into the fixed `padbuf[96]` (`ckcmai.c:844`).
`s[3]` is a plain array index (no side effect), so it is safe to bound there:

```c
    npad = xunchar(s[3]);
    /* [V-3] Valid pad count is 0..94.  A corrupt/hostile field-3 byte wraps */
    /* (a control char decodes to 224..255) and would overrun padbuf[96], so */
    /* treat any out-of-range count as "no padding".                         */
    if (npad > 94) {
      npad = 0;
    }
```

A conforming peer encodes the pad count as `tochar(0..94)` → bytes `32..126`, which decode to `0..94`
and pass through unchanged. A hostile/corrupt control byte in field 3 decodes (via the restored
`xunchar`) to `224..255`, is caught by `> 94`, and is treated as "no padding" — closing the padbuf
overflow without the mod-256 information leak.

> **Note on V-1 and V-2:** neither depends on the `xunchar` clamp. V-1 (login length) has its own
> `if (len > LOGINLEN) return(-1)` guard and V-2 (`putsrv`) its own bound; both were applied
> independently in Phase 1 and remain effective. Reverting the clamp does **not** re-open them.

## 5. Verification

- **Build:** `make linux` — warning-clean (0 warnings/errors), full clean rebuild.
- **Format:** `clang-format ckcker.h | diff` and `... ckcfns.c | diff` both empty (stable).
- **Interoperability (the regression):** the loopback matrix now **PASSES in every combination** —
  `fix ↔ fix`, `fix-server ↔ main-client`, `main-server ↔ fix-client` — each returning the full
  18-line directory listing (vs. 0 lines / "Protocol error" before).
- **Security (V-3 still closed):** an exhaustive check of all 256 possible field-3 byte values
  confirms every legitimate encoded pad count (bytes `32..126` → `0..94`) passes through unchanged,
  and **no** value reaches the `padbuf[96]` loop with `npad ≥ 96` after the clamp (0 overflow cases).
- **Smoke:** `./wermit --version` → `10.0`; `echo exit | ./wermit` → exit 0.

---

## 6. Scope & impact

- **Severity of the regression:** high — it broke *all* Kermit-protocol file transfer and REMOTE
  commands in the `fix-20260708` build, not just `REMOTE DIRECTORY`. Any deployment of that build for
  actual file transfer would have been non-functional.
- **Not shipped to `main`:** the defect exists only on the `fix-20260708` branch (Phase 1 onward).
- **Files changed by this bugfix:** `ckcker.h` (revert macro), `ckcfns.c` (add `npad` clamp). No
  `ckcpro.w` regeneration needed. No interaction with the `SVR4`/`SVR3`/`ATTSV`/`BSD44`/`MACOSX10`
  chain.

## 7. Lesson

When "centralizing" a check into a widely-used expression macro, verify it evaluates its argument
**exactly once** — the codebase's `xunchar()`/`tochar()` idiom is routinely called as
`xunchar(*p++)`. A clamp that must reference its argument more than once belongs in a
single-evaluation **function**, or at the specific **point of use** (as done here for `npad`), not in
a multiple-evaluation macro.
