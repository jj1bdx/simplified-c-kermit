# C-Kermit 10.0 Beta.12 (simplified fork) — no-arg REMOTE SPACE hang against macOS

> **⚠️ REVERTED (2026-07-09, commit `6d3f1a4`) — see `BUGFIX_20260709_3.md`.**
> The `SPACMD` change described in §3 below was reverted the same day: the hang reported here is
> now attributed to the `spack()` short/long-packet framing defect fixed in commit `c6e9bbc`
> (`BUGFIX_20260709_2.md`), which — unlike the blocking-volume mechanism analyzed below — was
> actually observed in packet logs and explains the intermittency. The blocking-`df` *mechanism*
> documented in §2 is real and was verified by simulation, but no blocking volume was ever
> observed on `daimac`, and the workaround changed macOS behavior (single-filesystem instead of
> all-volumes output) to guard against a hazard that was never shown to have fired. `ckufio.c`
> once again uses the original upstream guard, and no-arg `REMOTE SPACE` on macOS/BSD reports all
> mounted volumes. The analysis below is kept for the record.

**Subject:** `REMOTE SPACE` *without arguments*, against a macOS server (`daimac`), reported to
hang the whole session ("communication hanged up"). With an argument it does not hang.

- **Branch:** `fix-20260708` (HEAD = `4a9a26d` + this fix)
- **Date:** 2026-07-09
- **Companion docs:** `BUGFIX_20260708.md` (`xunchar()` regression), `BUGFIX_20260708_2.md`
  (SPACE/makefile), `BUGFIX_20260708_3.md` (over-ssh investigation, Linux server — could not
  reproduce; flagged macOS as untested).

## TL;DR

1. **The hang is real but conditional, and macOS-specific in practice.** On macOS the no-argument
   `REMOTE SPACE` server command runs a plain **`df` over ALL mounted volumes**. On a Mac with
   external / Time Machine / network volumes, `df` blocks whenever one of those volumes is asleep
   or wedged (disk spin-up, hung SMB/NFS mount). While it blocks, the transfer never completes and
   **neither side ever times out** — the session sits silent indefinitely: the reported hang.
2. **Why only the no-argument form:** `REMOTE SPACE <dir>` uses `SPACM2 = "df "` + the directory —
   `df <dir>` stats a *single* filesystem and never touches the sleeping/stuck volumes.
3. **Root cause of the wrong `df`:** `ckufio.c` selects the intended BSD behavior
   `SPACMD = "pwd ; df ."` under `#ifdef BSD4` — but **no live target defines `BSD4`**, and none
   ever did for macOS (the chain is `MACOSX10 → MACOSX → BSD44`; `BSD44` implies `SVR4`, not
   `BSD4` — verified in the original tree at `5a8fc5a`). The branch has been dead code on every
   platform this tree supports, in upstream C-Kermit too. **Not a simplification regression.**
4. **Fix:** guard the BSD-style `SPACMD` with `BSD44` (macOS, FreeBSD, NetBSD, OpenBSD) so the
   no-argument command reports the current filesystem only (`pwd ; df .`), exactly mirroring the
   already-safe `SPACM2` path. Linux is unchanged (`df `).

## 1. Reproduction attempts

The reported interactive flow (`./wermit` → `ssh daimac` → `./wermit` → `server` → `Ctrl+\ C` →
`remote space`) was scripted with `expect` and run repeatedly against the real `daimac`
(macOS 26.5.2, arm64, Darwin 25.5.0) over real ssh:

| Test | Result |
|---|---|
| exact reported flow, `remote space` (no arg) | PASS — full `df` output, fast |
| same session: `remote space` ×3 + `remote directory` | PASS |
| `remote space /tmp` etc. | PASS |

`df` on daimac completed in ~3 ms at test time — **the trigger condition (a blocking volume) was
not present**, so the hang did not reproduce directly. daimac mounts four external volumes
(`dock_base`, `tm_daimac2` (Time Machine), `daimac-essd-1`, `bigmac-extarc`) — a sleeping disk or
a stalled Time Machine/network mount makes plain `df` block for seconds to forever; `df` cannot be
interrupted while a volume is in that state.

## 2. The hang mechanism, verified

The blocking condition was simulated over the real daimac connection with
`remote host (sleep 45 && date)` and `remote host sleep 200; …` — `REMOTE HOST` uses the identical
server machinery as `REMOTE SPACE` (`syscmd()` → `zxcmd()` fork/pipe → `ssinit`/`sdata()` send
loop), with the block guaranteed instead of volume-dependent.

**Observed (timestamps + client packet log):**

- Command sent 01:20:44, output arrived 01:21:30 — the client sat in the transfer display
  ("Press the X or E key to cancel") for the full 46 s **without any timeout, NAK storm, or
  retry exhaustion**, then completed normally. Same for the 200 s variant.
- The packet log shows why neither side times out: while the child produces no output, the server
  emits a tiny **empty Data packet about once per second** (`zinfill()` polls the pipe via
  `ttwait()`/`select()` with a 1-second cap when `srvping` is on — the default — and `sdata()`
  turns the `-3` "no data yet" return into a zero-length D packet); the client ACKs each one
  (`<rdpkt>D` treats an empty D packet as valid traffic). The protocol stays fully synchronized —
  a **live wait with no cumulative cap on either side**. If `df` never finishes, this continues
  forever, silently.
- Recovery: the wait is architecturally cancelable client-side (`X`/`E` keys — `chkint()` runs per
  packet on the local end), and the session continues normally once the command completes. But
  nothing on screen says a cancel is possible, so in practice it is perceived as a dead session.

So: **not** a protocol/parsing defect (unlike the 20260708 `xunchar()` regression), and no
desync — an unbounded, invisible wait on a shell command that can legitimately never return.

## 3. The fix

`ckufio.c` (SPACMD selection): change the dead `#ifdef BSD4` guard to `#ifdef BSD44`, with a
comment explaining the hazard. All live BSD-family targets define `BSD44` (macOS via
`MACOSX10 → MACOSX → BSD44` in `ckcdeb.h`; FreeBSD/NetBSD/OpenBSD via `-DBSD44` in their makefile
targets). The tree is a pure-`#ifdef` tree, so no `#if defined(...) || defined(...)` form is used.

```c
#ifdef BSD44
char *SPACMD = "pwd ; df ."; /* Space in current directory */
#else
char *SPACMD = "df ";
#endif /* BSD44 */
```

- `pwd ; df .` touches only the current filesystem — sleeping/stuck volumes are never stat'ed,
  removing the hang trigger for the no-argument command.
- This is the code's own original intent for BSD systems (the `BSD4` branch encoded exactly this
  string); the guard was simply attached to a macro no supported system ever defined.
- `SPACM2` (`df <dir>`) is unchanged; Linux (`SPACMD = "df "`) is unchanged.
- User-visible change on macOS/BSD: no-arg `REMOTE SPACE` now prints the working directory and one
  `df` line for it, instead of a table of every mounted volume.

## 4. Verification

- **Linux:** `make linux` — warning-clean; `SPACMD` unchanged there (`BSD44` undefined).
- **macOS (daimac):** same edit applied, `make macos` — warning-clean.
- **End-to-end over real ssh** (fixed client ↔ fixed macOS server): no-arg `remote space` returns
  the `pwd` line + single-filesystem `df` immediately; `remote space /tmp` (SPACM2 path) and
  `remote directory` (regression check) both pass; `finish` clean.
- `clang-format ckufio.c | diff` — the edited region is format-stable (the two whitespace-unstable
  comment hunks at lines ~5139/~6030 pre-exist at HEAD, unrelated).

## 5. Out of scope / residual notes

1. **The unbounded server wait is still there, generically.** Any `syscmd()`-based server command
   (`REMOTE HOST`, `WHO`, `DIRECTORY`, `TYPE`, `PRINT`, and `SPACE <dir>` against a stuck
   filesystem) still waits forever if the child never produces output — with the same silent
   1 Hz empty-packet keepalive. A cumulative cap (give up + `kill()` the child after N minutes of
   no output) would be defense-in-depth, but it means surgery in the shared
   `sdata()`/`zinfill()`/`zxcmd()` polling logic used by every such command — deliberately not
   attempted here.
2. **Sibling dead branch:** `WHOCMD` in `ckufio.c` has the same dead `#ifdef BSD4` (`"finger "` vs
   live `"who "`). Left untouched — `who` works fine on macOS and has no blocking hazard.
3. **Linux plain `df`** is equally exposed to a hung NFS/CIFS mount in principle; left as-is to
   keep the change minimal and matching the report.
