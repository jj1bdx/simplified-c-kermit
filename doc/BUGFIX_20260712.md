## 1. Bug 1 — hostile/one-sided CHKT `'5'` in a Y ACK

### Root cause and reachability

`spar()` (`ckcfns.c`) accepts CHKT field values 1..5 into `bctr`. The `'5'`
("force block check 3") convention is only *recognized* on incoming I/S packets —
`rpack()` (`ckcfn2.c`) sets `bctf` when `recpkt[11] == '5'` for `type=='I'||'S'` — never
on a Y. So a `'5'` in an ACK leaves `bctf==0`, and `ckcpro.w`'s

```c
if (bctf) { bctu = 3; bctl = 3; } else { bctu = bctr; bctl = (bctu == 4) ? 2 : bctu; }
```

produces `bctu = bctl = 5`. `spack()`'s block-check `switch (bctu)` had cases 1..4 and
**no default**: every outgoing packet then carried a LEN that counted 5 check bytes that
were never appended — unparseable by any receiver. (The receive side was already safe:
`rpack()`'s check switch has a `default:` returning `'Q'`.)

**This is not hostile-peer-only** (c-expert finding, verified): `rpar()` builds both our
S/I packets *and* our ACKs to them, and emits `bctr + '0'` unconditionally. A peer whose
user typed `SET BLOCK 5` — while *our* user did not — puts a literal `'5'` in its ACK to
our Send-Init. One-sided ordinary configuration, no malice needed.

### Fix

- `ckcfns.c` `spar()`: after the existing 1..5 range clamp, `if (x == 5 && !bctf) x = 3;`
  — type 3 matches both what a genuine force-3 peer expects in the data phase and the
  in-tree precedent (the REMOTE SET BLOCK-CHECK handler, case 400, already maps an
  incoming `'5'` to `bctr = 3`). We deliberately do **not** set `bctf` from an ACK
  field: `bctf` is never reset anywhere once set, so adopting it from one suspicious
  packet would degrade the rest of the process lifetime.
- `ckcfn2.c` `spack()`: new `default:` that debug-logs and returns -1 (same failure
  contract as the 6b36bb7 short-packet guard) — belt-and-braces, unreachable after the
  clamp.

### Verification

Hostile peer: throwaway worktree build (6b36bb7 + `rpar()` forced to emit `dada[7]='5'`)
as server; device under test as client sending a 200-byte file.

- pre-fix client: `SPACK bctl=5` in the debug log, transfer FAIL (exit 3).
- fixed client: transfer PASS, received file byte-identical; debug log shows the data
  phase at `SPACK bctl=3` (5 packets) with only the pre-negotiation I/S exchanges at
  `bctl=1` — the clamp engaged and interoperated.

## 2. Bug 2 — `getpkt()` memory-string overrun (and the stale-`t` continuation bug)

### Root cause

`getpkt()`'s overflow handler treats file sources and memory strings differently: file
sources truncate the packet at `odp` (start of the last encoded sequence) and carry the
excess in `leftover[]`; the memstr branch returned the **overfull** packet as-is (up to
one full prefixed/repeat-doubled sequence past `bufmax`) after un-reading one lookahead
char. `memstr` has exactly two producers:

- `encstr()` — one-shot (filenames, REMOTE command strings, E-packet text, X-packet
  headers). Overrun mattered only when the *final* sequence straddled `bufmax`: the
  string was fully consumed, so `encstr()`'s completeness check reported success on an
  overlong field. Against short-packet sessions the 6b36bb7 `spack()` guard already
  converted that into a refusal; in LP sessions the packet exceeded the peer's stated
  `rpsiz` by a few bytes.
- `sndstring()` — **multi-packet** streaming of a memory string (the long-form
  `REMOTE QUERY` reply). Here a second, unrecorded bug surfaced during analysis: every
  other `getpkt()` early-return path threads the pending lookahead char through the
  file-static `t` (`t = rt;`), but the memstr branch returns before that line, while
  the resume path `register CHAR rt = t;` still consults it when `first==0`. Each
  packet boundary therefore *inserted one stale byte* into the stream — and combined
  with the overfull return against a non-LP peer, the reply's oversized packet was
  unencodable and hung the transaction outright.

`funcstr` sources (`REMOTE DIRECTORY/SPACE/TYPE/...`) use the file-style `leftover[]`
path and were never affected.

### Fix (`ckcfns.c`, `getpkt()`)

Per-iteration bookkeeping (near-zero cost — two int stores per character):

- `seqchars` — how many *source* chars the encoded bytes `[odp,dp)` represent:
  1 normally; the repeat count at both `xxrc` emission points (the `rpt==94` dump and
  the run-end `tochar(++rpt)`); 2 in the `rpt==1` doubling block, downgraded back to 1
  when the block moves `odp` to `p2` (then only the second copy is beyond `odp`).
- `lsstate_save` — snapshot of the locking-shift state, restored if the sequence is
  dropped; refreshed inside the doubling block's `odp`-move (the shift char went out
  with the *kept* first copy, so it must not be rolled back then).

New memstr overflow branch:

- **Overfull** (`dp-data > bufmax`): truncate at `odp`, `memptr -= seqchars` plus one
  more for this iteration's own lookahead fetch (a real char or the terminating NUL —
  always exactly one `*memptr++`), roll back `lsstate`, and set `first = 1` so the next
  call **re-primes from `memptr`** — stale `t` is never consulted again.
- **Exact fit with a char pending**: un-read it (as before) and set `first = 1` — this
  alone fixes the stale-byte insertion for multi-packet strings, and keeps `encstr()`'s
  "string not fully encoded" check working.
- **Exact fit at true EOF**: unchanged (`first` is already -1).
- **KANJI/JEUC** (`xlate && tcharset == TC_JEUC`): `zkanji()`/`kgetm()` buffer pending
  bytes and shift state internally with no fixed bytes-per-char relation to `memptr`,
  so rewind arithmetic cannot apply; the original behavior is preserved verbatim for
  that combination (pre-existing, reachable only via `sndstring()` + JEUC transfer
  charset).

### Verification

`REMOTE QUERY KERMIT repeat(abcdefghij,30)` (evaluated server-side → 300 chars, no RLE
runs) with the client in non-LP mode (~89-byte data fields → 4 packets, 3 boundaries):

- baseline (c77999d) server: transfer stalls, client killed at 40 s (the overfull packet
  is unencodable without long packets).
- fixed server: PASS — the client receives the exact 300-byte value; same query over a
  normal LP session also PASS (single-packet path regression).

## 3. Bug 3 — batch `SEND *` and unsendable files

### What the pre-fix behavior actually was (worse than documented)

Live repro (non-LP peer, three files, one with an RLE-incompressible 120-char name —
note: a 120×`'L'` name does *not* repro; repeat-count encoding collapses runs):

- **Long-named file sorts first** → the `<ssinit>Y` state gets `sfile() == 0` and
  aborts the **whole batch**: nothing transfers, and the error is the misleading
  fallback `"Can't open file"` (from empty `epktmsg`).
- **Long-named file mid-batch** → the `<sseof>Y` loop skipped it silently, **and every
  file after it was silently lost too**: the 6b36bb7 encstr-guard in `sfile()` returns
  after `openi()` succeeded, leaking the open input channel, so each subsequent
  `sfile()`→`openi()` failed. (This was a regression introduced by 6b36bb7 —
  pre-6b36bb7 the same case hung the session instead, so it was never a *working*
  path, but the leak is 6b36bb7's.)

### Fix

- `ckcfns.c` `sfile()`: on the encstr-failure path, close the input file exactly as
  `clsif()` would (`memstr`/`zclose(ZIFILE)`, reset `i_isopen`) but without its
  statistics/display side effects, and set
  `epktmsg = "Filename too long for packet length"` so the `<ssinit>Y` abort message
  is accurate.
- `ckcpro.w` `<sseof>Y` loop: on `sfile()` failure, announce the skip —
  `xxscreen(SCR_FN,...)` **then** `xxscreen(SCR_ST,ST_SKIP,SKP_TLN,...)` (the `SCR_FN`
  call loads the filename buffer the `ST_SKIP` display reads; same paired-call pattern
  `gnfile()` uses for `SKP_ACC`), plus a transaction-log line. Also short-circuit the
  loop when `fatalio` is set: with a dead connection every remaining `sfile()` fails
  identically, and the old loop would grind through the entire remaining batch before
  failing.
- New skip-reason code `SKP_TLN` (`ckcker.h`) with `skreason[]` entry
  `"Name too long for packet"` (`ckuusx.c`).
- `<ssinit>Y` deliberately keeps its abort-on-first-file semantics (longstanding
  upstream behavior for an unsendable first file) — now with an accurate message.

### Verification

- mid-batch: pre `first=OK last=MISSING`, silence; post `first=OK last=OK`, long file
  absent, and — driven through a real pty, since the transfer display is hard-disabled
  when stdio is redirected (`conbgt()` recomputes `backgrd` from `isatty()` and ignores
  `SET BACKGROUND`) — the brief display shows
  `SEND <name>: SKIPPED (Name too long for packet)`.
- first-in-batch: pre `"Fatal Kermit Protocol Error: Can't open file"`; post the same
  clean abort but with `"Filename too long for packet length"`.

## 4. Bug 4 — receive-side silence on corrupt-LEN packets

### Root cause (mechanism confirmed end-to-end)

`ttinl()` (`ckutio.c`) returned -1 for two garbage-LEN cases (control char, > 94) and
for an out-of-range SEQ. `rpack()` treats -1 as a **timeout** (`'T'`), and `input()`'s
receive-side `'T'` handler does `if (ttchk() > 0) continue;` — don't NAK while input is
pending. But the malformed-header returns fire at the LEN/SEQ byte, i.e. while the
packet's tail is still arriving, so input *is* essentially always pending: the NAK was
withheld every time, the sender retransmitted the identical malformed packet on its own
timeout, and the session sat in a silent mutual wait. Two of the three early returns
also skipped `ttimoff()`, leaving the SIGALRM timer armed across the return (latent
longjmp-into-stale-frame risk; the bad-SEQ path already disarmed it). A side casualty:
with `SET SERVER IDLE-TIMEOUT` armed, a corrupt packet's `'T'` could count as an idle
timeout and make the server exit.

### Fix

- `ckutio.c` `ttinl()`: the three malformed-header paths (garbage LEN, rubout LEN, bad
  SEQ) now disarm the alarm (`if (timo) ttimoff();`) and return **-4**, a new code
  distinct from -1 timeout / -2 interrupt / -3 fatal. (`ttinl()`'s only callers are the
  three `rpack()` variants; the genuine-timeout and streaming -1 returns are untouched.)
- `ckcfn2.c` `rpack()`: `j == -4` → free the buffer, log `<crunched:hdr>`, and return
  `'Q'` — the existing crunched-packet code that `input()` answers with an immediate
  NAK on both receive and send sides. **Ordering matters**: the check sits before the
  `j < -1` fatal branch, which would otherwise swallow -4 as "connection lost" and
  abort the transfer — strictly worse than the hang (c-expert finding).
- Streaming behavior is unchanged (`'Q'` and `'T'` are already both fatal on reliable
  links), and the `srvidl` false idle-exit can no longer trigger from a malformed
  packet.

### Verification

Baseline (c77999d) sender — which emits a 90-byte D packet as LEN `tochar(95)` = DEL
against a non-LP CRC-16 receiver — vs. the fixed receiver:

- pre-fix receiver: packet log shows `<timeout>` entries only, **zero NAKs**, client
  killed at 60 s with no error ever surfacing.
- fixed receiver: `<crunched:hdr>` logged and a NAK sent immediately on each corrupt
  packet (three exchanges within the first second), and the sender abandons the
  transfer with a visible "Protocol error" instead of waiting forever. (Subsequent
  NAKs pace at the *sender's* retransmission timeout, and their sequence numbers march
  forward — that is `input()`'s existing "NAK the oldest un-NAK'd" policy interacting
  with the asymmetric window sizes of the `negotiation-string-max-length 8` simulation,
  not new behavior.)

Both fixed instances of this fork cannot produce such packets in the first place
(6b36bb7 + this change), so this path now only defends against *other* implementations.

## 6. Analysis-process notes

c-expert ran two rounds. Round 1 was strong on bugs 1/3/4 (it found the legitimate
one-sided `SET BLOCK 5` trigger, the `bctf`-is-never-reset argument against adopting
forcing from an ACK, the `SCR_FN`-before-`ST_SKIP` display pairing, the `fatalio`
batch-spin, and the -4-before-`j<-1` ordering trap) and it independently discovered the
stale-`t` memstr continuation bug. Its round-1 bug-2 patch had three defects, each
confirmed by trace in round 2:

1. its `t = rt` addition, combined with the `memptr--` rewind, would have **encoded the
   pending char twice** at every continuation boundary (the resume path reads from both
   `t` and the rewound `memptr`); replaced by `first = 1` re-priming so the pending
   char lives in exactly one place;
2. its doubling-block `seqchars = 2` ignored that the block *moves* `odp` when the
   first copy fits (dropped unit = 1 char, and the locking shift goes out with the
   kept copy, so `lsstate` must not be rolled back then);
3. it initially missed that `KANJI` is compiled in (defined unconditionally via
   `ckcxla.h` — verified with `cc -E -dM` under the real `make linux` flags, the same
   transitive-definition trap as `NEWDEFAULTS`), which makes the JEUC gate necessary.

Round 2 also contributed the unification of the two overfull sub-cases into one
formula (`memptr -= seqchars; memptr--;` — every loop iteration performs exactly one
fetch, whether of a real char or the terminating NUL).

Empirical surprises during repro work: a 120×`'L'` filename does not trigger the
long-name failure (RLE collapses it — incompressible names required); the transfer
display is unconditionally suppressed when stdio is redirected (`conbgt()` recomputes
`backgrd` from `isatty(0/1)` on every `ckscreen()` call and `SET BACKGROUND OFF` cannot
override it), which is why the skip-message check needed a real pty.

## 7. Out of scope / follow-ups

1. **`ack()`/`ack1()` return values are discarded** at ~25 call sites in `ckcpro.w`; a
   `spack()` refusal inside an ACK would advance the state machine with nothing sent
   (the same "phantom success" class fixed for `sfile()` in 6b36bb7). Dead code today —
   both guards are unreachable after the `spar()` clamps — but worth an audit if more
   `spack()` refusal paths are ever added. Likewise `nack()` conflates "too many
   retries" with a `spack()` failure in its -1.
2. **Priming/loop `fmask` inconsistency**: the `first==1` priming block reads memstr
   chars without applying `fmask`, the main loop applies it; and a byte that masks to
   zero (`SET FILE BYTESIZE 7` + a 0x80 char) is silently dropped at an exact-fit
   boundary (pre-existing; unchanged by this fix). One-line follow-up if BYTESIZE-7
   memstr transfers ever matter.
3. **`recpkt[11]` bctf sniffing in `rpack()`** reads a fixed offset without checking
   the packet is long enough for a CHKT field; a short S packet whose block-check byte
   lands at offset 11 as `'5'` could set `bctf` spuriously. Not touched.
4. **`<ssinit>Y` batch abort semantics**: when the *first* file of a batch is
   unsendable the whole batch still aborts (now with an accurate message). Skipping to
   the next file instead would need the `<sseof>Y` gnfile-loop replicated there;
   deliberate scope cut.
5. **macOS build** re-verification on daimac (`make macos` + a loopback smoke test), as
   for previous protocol changes.
