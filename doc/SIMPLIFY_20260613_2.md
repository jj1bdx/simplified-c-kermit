# Codebase Simplification Report — Removal of non-ANSI / K&R C Code Paths

**Date:** 2026-06-13
**Tree:** C-Kermit 10.0 Beta.12 (`/home/kenji/src/simplified-c-kermit`)
**Scope:** Assume the compilation macro `CK_ANSIC` is *always* defined — the codebase
now only ever targets ANSI/ISO C compilers. Remove the dead non-ANSI / K&R C
branches gated by `CK_ANSIC` (and its inverse switch `NOANSI`), keeping only the
ANSI prototype / `void` / `const` paths. No platform code (Linux/macOS/Unix) and no
runtime behaviour is affected; this is purely the deletion of pre-ANSI fallback code
that no supported compiler ever selects.

## Method

1. **Analysis.** `CK_ANSIC` appears in ~40 files; `NOANSI` (its inverse) in 2
   (`ckcdeb.h`, `ckcfnp.h`). The dominant pattern is dual-form function definitions:

   ```c
   int
   #ifdef CK_ANSIC
   foo(char *buf, int len)
   #else
   foo(buf,len) char *buf; int len;
   #endif
   { ... }
   ```

   plus the macro *definitions* in `ckcdeb.h` that select ANSI vs K&R spellings of
   `_PROTOTYP`, `VOID`, `CKVOID`, `CONST`, and the `debug`/`ckhexdump` macros, and the
   `ckcfnp.h` prototype file wrapped in `#ifndef NOANSI / #ifdef __STDC__ / #ifdef
   CK_ANSIC`.

2. **Conditional-block removal** with `/usr/bin/unifdef -m -DCK_ANSIC -UNOANSI`
   (in-place), over all `*.c *.h *.w` **except the generated `ckcpro.c`**, with
   per-file return-code checking (`unifdef` exits 0 = unchanged, 1 = changed, 2 =
   error). Result: 39 changed, 12 unchanged, **0 errors**, **0 parse problems**.
   Treating `-DCK_ANSIC` keeps every ANSI branch and drops every K&R branch;
   `-UNOANSI` collapses the `NOANSI` guards (`NOANSI` is the "disable ANSI" switch and
   is, by the new assumption, never defined).

3. **Regeneration of `ckcpro.c`** from the cleaned `ckcpro.w`
   (`make wart && make ckcpro.c`).

4. **Build + smoke test** on Linux (`rm -f wermit && make clean && make linux`, then
   `make check`).

### Why `unifdef` was safe here (no `-D` traps)

Unlike a feature removal, this change has a clean polarity: `CK_ANSIC` is set true
and `NOANSI` false, and the two are never combined with other symbols in compound
`#if defined(A) || defined(B)` expressions. So `unifdef` performed pure, mechanical
block selection. The one place where `CK_ANSIC` is *nested under* an unknown symbol —
`ckcdeb.h` and `ckcfnp.h`, where the prototype includes are guarded by
`#ifndef NOANSI / #ifdef __STDC__ / #ifdef CK_ANSIC` — was handled correctly:
`unifdef` removed the `NOANSI` and `CK_ANSIC` layers and **left the `#ifdef __STDC__`
guard intact** (it is a real compiler-defined symbol, not one we are forcing). The
prototype block therefore remains correctly conditioned on the compiler actually
being a standard C compiler, which is the original intent.

## Macro-definition vs macro-invocation decision

The `#ifdef CK_ANSIC … #else … #endif` blocks in `ckcdeb.h` that *define* the
portability aliases were reduced by `unifdef` to their ANSI branch only, and the
defines are now **unconditional**:

| Macro | Now (unconditional) | Was (K&R `#else`) |
|---|---|---|
| `_PROTOTYP(func,parms)` | `func parms` | `func()` |
| `VOID` | `void` | `int` |
| `CKVOID` | `void` | `int` |
| `CONST` | `const` | *(empty)* |
| `debug(a,b,c,d)` | `((void)(deblog?dodebug(...):0))` | `(deblog?dodebug(...):0)` |
| `ckhexdump(a,b,c)` | `((void)(deblog?dohexdump(...):0))` | `(deblog?dohexdump(...):0)` |

The corresponding prototype helpers `ckxprintf`/`ckxperror`/`ckxfprintf` (formerly
guarded by `#ifdef CK_ANSIC`) and the `ckcfnp.h` include are likewise now selected
unconditionally (the latter still under `#ifdef __STDC__`).

**Decision: the macro *invocations* throughout the tree were deliberately NOT
expanded.** `_PROTOTYP(...)`, `VOID`, `CKVOID`, and `CONST` remain in use at thousands
of call sites (e.g. `_PROTOTYP( int foo, (char *, int) )`, `VOID doexit()`,
`CONST char *p`). Now that their definitions are fixed one-line aliases, these
invocations are harmless, fully valid, and identical in meaning to their expansions.
Rationale for keeping them:

- Expanding them would be a massive, mechanical, risky edit (every signature in
  `ckcfnp.h` plus thousands of definitions and declarations across 40+ files) for
  **zero** functional or correctness benefit.
- `_PROTOTYP` in particular spans multi-line declarations with inline `/* */`
  parameter-name comments; a naive expansion would mangle formatting.
- The aliases are self-documenting and consistent with the project's deliberately
  conservative, portable style described in `CLAUDE.md`.

The goal of this task was to **remove the dead non-ANSI branches**, not to rewrite the
codebase. The dead branches are gone; the aliases stay, now defined unconditionally.

## Risks / notes (constructs requiring care)

- **No hand-fixes were needed.** `unifdef -m` produced a clean tree that built with
  zero warnings on the first attempt; no dangling `#else`/`#endif` or orphaned
  declarations were left behind.
- **Inert residual references to `CK_ANSIC`/`NOANSI`** remain and were intentionally
  left untouched, as none is an active code path:
  - `ckcdeb.h` (comment block documenting the now-unconditional `_PROTOTYP` macro and
    the historical `-DCK_ANSIC` / `-DNOANSI` command-line switches).
  - `ckctel.c`, `ckcnet.c` (author comments such as "don't use any ANSI C constructs
    except within `#ifdef CK_ANSIC..#endif`", and a `-d=CK_ANSIC` build-command
    example in a comment).
  - `ckuus5.c` (`makestr(&(optlist[noptlist++]),"CK_ANSIC");`) — this is the
    `SHOW FEATURES` runtime option list. It was formerly inside
    `#ifdef CK_ANSIC … #endif`; `unifdef` correctly removed the guard and kept the
    `makestr` unconditional, so C-Kermit still reports `CK_ANSIC` as an enabled build
    feature. Inert string literal, intentionally preserved.
- **Empty `#ifdef __STDC__ / #endif` pair** at `ckcdeb.h` (formerly the self-defining
  `#define CK_ANSIC` site). Left as-is: `__STDC__` is a genuine compiler symbol and
  out of scope for this `CK_ANSIC`-only cleanup; the empty guard is harmless.
- **`ckcpro.w` was edited** (4 lines), so `ckcpro.c` was regenerated via
  `make wart && make ckcpro.c` before building (it is gitignored / not committed).
