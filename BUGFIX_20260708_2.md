# C-Kermit 10.0 Beta.12 (simplified fork) — Regression Analysis & Fix (part 2)

**Subject:** `REMOTE SPACE` reported failing between `fix-20260708` and `main`, and `fix ↔ fix`.

- **Branch:** `fix-20260708`
- **Date:** 2026-07-08
- **Companion docs:** `BUGFIX_20260708.md` (the `xunchar()` regression), `VULNERABILITIES-20260707.md`, `FIXES-20260708.md`.

---

## TL;DR

1. **`REMOTE SPACE` is not a new bug.** It was broken by the *exact same* `xunchar()`
   multiple-evaluation regression (V-3) that broke `REMOTE DIRECTORY`, and it is **already fixed** by
   commit `7ec09ca` (`BUGFIX_20260708.md`). `REMOTE SPACE` and `REMOTE DIRECTORY` share the same
   packet receive/decode path (`rpack()` → `decode()`), so a bug there breaks both.
2. **It works at HEAD.** Verified by loopback: `REMOTE SPACE` **fails** on the pre-fix commit
   (`1356430`) and **works** on `7ec09ca` across every client/server version combination.
3. **One real, separate defect was found and fixed:** the **makefile omitted `ckcker.h`** from the
   dependency lists of four objects that `#include` it (`ckclib`, `ckucmd`, `ckufio`, `ckutio`). On
   an *incremental* `make linux` after editing `ckcker.h`, those objects were **not recompiled** — a
   stale-object hazard that can make a "rebuilt" tree behave inconsistently with the source. This is
   the most likely reason a rebuild might not fully reflect the `xunchar()` fix.

**Action required:** rebuild `fix-20260708` at HEAD — ideally `make clean && make linux` (a plain
incremental `make linux` also now suffices after the makefile fix below).

---

## 1. Reproduction

Two `wermit` instances bridged over a `socat` PTY pair (one `server`, one issuing `remote space`),
with `log packets`/`log debug`. `REMOTE SPACE` runs a shell command (`SPACMD`=`"df "` on Linux;
`"pwd ; df ."` on BSD/macOS, via `SPACM2` for a specified area) and streams the output back as a
text transfer — the same send/receive machinery as `REMOTE DIRECTORY`.

| Build | `remote space` | `remote directory` |
|---|---|---|
| **pre-fix** `1356430` (before the `xunchar()` fix) | **FAIL** (0 lines, "Protocol error") | **FAIL** (0 lines) |
| **HEAD** `7ec09ca` (with the `xunchar()` fix) | **PASS** (full `df` output) | **PASS** (full listing) |

At HEAD, `REMOTE SPACE` passes in **all** combinations — `fix ↔ fix`, `fix-server ↔ main-client`,
`main-server ↔ fix-client`, `main ↔ main` — with and without a path argument
(`remote space`, `remote space /home`, `remote space .`, `remote space /nonexistent`).

> Note: a trailing "Protocol error" appears in a test only if the client script ends with `bye`
> while the server has `BYE` disabled ("BYE disabled") — that is a test-script artifact, identical on
> `main`, and unrelated to `REMOTE SPACE`. Ending the client with `finish` instead gives clean output.

---

## 2. Root cause (same as `BUGFIX_20260708.md`)

`REMOTE SPACE`, like every Kermit-protocol operation, decodes incoming packets through `rpack()`
(`ckcfn2.c`) and `decode()` (`ckcfns.c`), both of which use the `xunchar()` macro. V-3 had redefined
`xunchar()` as a clamping ternary that **evaluates its argument up to three times**; because callers
pass side-effecting expressions (`xunchar(recpkt[i++])`, `xunchar(*xdbuf++ & 0xFF)`), the packet
length/sequence/checksum reads advanced their index 2–3× and every packet was mis-parsed → NAK →
"Protocol error". Commit `7ec09ca` restored single-evaluation `xunchar()` and moved V-3's real
protection (the pad-count clamp) into `spar()`. That fix closes `REMOTE SPACE` and `REMOTE DIRECTORY`
together; there is nothing SPACE-specific to fix.

---

## 3. Why it can still look broken after "updating" — the makefile dependency gap

Investigating "why would a rebuilt tree still fail" turned up a genuine build bug. Four source files
`#include "ckcker.h"` but their makefile rules did **not** list it as a prerequisite:

```
ckclib.$(EXT):  ckclib.c ckclib.h            ... ckcfnp.h   # no ckcker.h
ckucmd.$(EXT):  ckucmd.c ckcasc.h ckucmd.h   ... ckcfnp.h   # no ckcker.h
ckufio.$(EXT):  ckufio.c ckcdeb.h ckuver.h   ... ckcfnp.h   # no ckcker.h  (uses xunchar at :5042)
ckutio.$(EXT):  ckutio.c ckcdeb.h ckcnet.h   ... ckcfnp.h   # no ckcker.h  (uses xunchar at :8266+)
```

Consequence: after editing `ckcker.h` (as the `xunchar()` fix did), an **incremental** `make linux`
would recompile `ckcfn2.o`/`ckcfns.o`/`ckcpro.o` (which *do* list `ckcker.h`) but leave
`ckclib.o`/`ckucmd.o`/`ckufio.o`/`ckutio.o` **stale**, compiled against the old header.

Impact on the `REMOTE SPACE` symptom: **none, in practice.** The SPACE/DIRECTORY decode path is in
the correctly-tracked objects, and the `xunchar()` uses in `ckufio.c`/`ckutio.c` take side-effect-free
index arguments (`xunchar(dest[1] & 0x7f)`, `xunchar(*(yy->gprotect.val))`), so they were never
corrupted by the multi-eval bug. But it is a real reliability hazard: an incremental build after a
header change can silently ship stale objects, which is exactly the class of "I rebuilt and it's
still wrong" confusion this report reflects. It should be fixed regardless.

---

## 4. The fix

Add `ckcker.h` to the four rules that were missing it, so an incremental build recompiles every
object that includes it when it changes:

```make
ckclib.$(EXT): ckclib.c ckclib.h ckcker.h ckcdeb.h ckcasc.h ckcfnp.h
ckucmd.$(EXT): ckucmd.c ckcasc.h ckucmd.h ckcker.h ckcdeb.h ckctel.h ckclib.h \
		ckcfnp.h
ckufio.$(EXT): ckufio.c ckcker.h ckcdeb.h ckuver.h ckclib.h \
		ckcxla.h ckuxla.h ckcuni.h ckcfnp.h
ckutio.$(EXT): ckutio.c ckcker.h ckcdeb.h ckcnet.h ckuver.h ckctel.h ckclib.h \
		ckcfnp.h
```

This is purely additive (it can only cause *more* correct recompilation) and does not change the
build for a clean tree. No source change is needed — the protocol fix already lives in `7ec09ca`.

## 5. Verification

- **Dependency audit:** every `.c` that `#include`s `ckcker.h` now lists it in its makefile rule
  (0 remaining gaps).
- **Build:** `make linux` — warning-clean (0 warnings/errors).
- **Recompile trigger:** `touch ckcker.h` now schedules `ckclib.c`, `ckucmd.c`, `ckufio.c`,
  `ckutio.c` for recompilation (previously it did not).
- **Incremental-build proof:** starting from a tree built at `1356430`, applying only the `xunchar()`
  source fix and running a plain incremental `make linux` produces a binary whose `REMOTE SPACE`
  works (full `df` output) — confirming the fix does reach the SPACE path once the correct objects
  are rebuilt.
- **End-to-end:** `REMOTE SPACE` returns the correct `df` output across `fix↔fix`, `fix↔main`,
  `main↔fix` at HEAD.

## 6. Action items

1. **Rebuild at HEAD.** `git checkout fix-20260708 && make clean && make linux`. This guarantees all
   objects reflect the `xunchar()` fix. (`REMOTE SPACE` and `REMOTE DIRECTORY` both work afterward.)
2. **Apply the makefile fix** (this commit) so future incremental builds after a `ckcker.h` edit are
   reliable and cannot ship stale objects.
3. If `REMOTE SPACE` still fails after a clean rebuild at HEAD, the cause is environmental (e.g.
   `df`/`push` disabled on the server → server replies "Can't check space", or `REMOTE SPACE disabled`
   → "REMOTE SPACE disabled") rather than a protocol/parsing defect — those are clean, expected error
   replies, not communication failures.

## 7. Note

macOS/BSD builds use `SPACMD = "pwd ; df ."`; the `xunchar()` fix is platform-independent (a header
macro), so `REMOTE SPACE` is fixed identically there. The makefile fix likewise affects all platform
targets, which share these object rules.
