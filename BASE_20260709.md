### Recurring hardening idioms

The fixes cluster into a handful of patterns rather than ad-hoc edits:

- **Bounded-cursor helper/macro replacing raw `*p++ = c`** — `RPBUFPUT()` in `ckcfn3.c` (V-4),
  `bufcpy()` in `ckucmd.c` (V-15), `TOKVALMAX` bound on the `.netrc` loops in `ckcftp.c` (V-17),
  the `srvptr` bound in `putsrv()` (V-2).
- **Format-string elimination** — hand-rolled `%s`-only substitution helpers `dialexpand()`
  (`ckudia.c`, V-20) and `pctsexpand()` (`ckuusr.c`, V-27) replace `sprintf(buf, userstring, arg)`,
  so a config/attacker-influenced string is never used as a `printf` format.
- **Array-index / table-lookup clamping** — the `spar()` pad-count clamp `if (npad > 94) npad = 0;`
  (`ckcfns.c`, the relocated V-3), `if (c & 0x80) return (UNK);` in `xeglg()` (`ckuxla.c`, V-26),
  and lower-bound self-guards in `dgi_u()`/`hproman8_u()` (`ckcuni.c`, V-38).
- **Open-flag TOCTOU/symlink hardening** — `creat()`→`open(…,O_EXCL,…)` for the serial lock file
  (`ckutio.c`, V-19); `fopen()`→`open(…,O_NOFOLLOW,…)`+`fdopen()` for file writes (`ckufio.c`, V-22).
- **Path-traversal lexical guards** — component-exact `..` checks (`..foo` is *not* flagged):
  `zhasdotdot()` in `ckufio.c` (V-6) and `has_dotdot()` in `ckcftp.c` (V-12).
- **`getenv()`/empty-string NULL and length guards, resource-leak closes, recursion caps** — V-37,
  V-36, V-21, V-28, V-40, V-41.

All of these are documented and verified to be **no-ops on well-formed input** — they change
behavior only for malformed, hostile, or truncated data.

## 2. The V-3 regression and its correction

V-3 (Phase 1) originally rewrote the `xunchar()` macro in `ckcker.h` as a clamping ternary. That
macro **evaluates its argument up to three times**, and nearly every caller passes a side-effecting
expression (`xunchar(*p++)`, `xunchar(recpkt[i++])`) — so pointers/indices advanced 2–3× per call
and **every Kermit packet parse broke** (checksum computed over the wrong bytes → NAK → "Too many
retries"). This broke *all* protocol operations, `fix↔fix` as well as `fix↔main`.

`7ec09ca` fixed the regression by **reverting `xunchar()` to its original single-evaluation form**
and **relocating** V-3's real protection to the one place a runaway value could overflow a fixed
buffer — the S/I-packet pad count in `spar()` (`ckcfns.c`: `if (npad > 94) npad = 0;`), where the
argument is a plain array index with no side effect. See `BUGFIX_20260708.md`.

> **Documentation caveat:** `FIXES-20260708.md`'s V-3 entry still describes the (now-reverted)
> clamping-ternary macro, because it was written before `7ec09ca`. The **net** state of `ckcker.h`
> at `base-20260709` is: `xunchar()` **byte-identical to `base-20260616`** (unclamped, single-eval),
> plus an explanatory comment. The ternary survives nowhere in the tree; `ckcfns.c`'s `spar()` clamp
> is where V-3 lives now.

## 3. Protocol / packet-layer changes

Two changes alter how the protocol state machine handles a packet (as opposed to bolting a bound
onto existing code):

- **`spack()` short/long-packet framing (`ckcfn2.c`, threshold `> 96` → `> 94`).** The short-packet
  LEN field is one printable character `tochar(len+bctl+2)`, legal only for 0..94 (`tochar(94)='~'`).
  The old threshold let totals of 95/96 take the short path and emit LEN bytes DEL(127)/0x80, which
  the receiver's `ttinl()` framer silently discards with no NAK — the sender then retransmits the
  identical unanswerable packet 33 times (~4.5 min) before `E "Too many retries"`. It fires whenever
  a transfer's final chunk lands on exactly 90/91 data bytes under block-check 3 (this fork's
  default) — pure data-size luck. This is an **upstream C-Kermit 10.0 Beta.12 bug**, present verbatim
  at the fork's initial commit — not a simplification regression. Fixed by routing those sizes
  through extended-length framing. This was the true root cause behind the reported "REMOTE SPACE
  hang" (see §4). Details in `BUGFIX_20260709_2.md`.
- **`srv_login()` oversized-username abort (`ckcpro.w`, V-1).** A REMOTE LOGIN username longer than
  `LOGINLEN` (33) sent an error packet but then fell through into an unconditional copy into a
  33-byte stack buffer (up to ~222 attacker-controlled bytes). Added `RESUME; return(-1);` so the
  server rejects the login and waits for the next command. (`ckcpro.w` is the protocol grammar;
  `ckcpro.c` is regenerated from it via `make wart && make ckcpro.c`.)

All other packet-path edits (V-2 `putsrv`, V-3 `spar` clamp, V-4 `RPBUFPUT`, V-29 truncated-escape
breaks) are pure hardening — no-ops on well-formed traffic.

## 4. The REMOTE SPACE "hang" investigation (and a reverted workaround)

A reported hang — no-argument `REMOTE SPACE` from a macOS client (`daimac`) against a Linux server
(`gluonite`) — was first attributed (`7f77234`, `BUGFIX_20260709.md`) to macOS's no-arg `SPACMD`
running an all-volumes `df` that can block on a sleeping/wedged volume, and worked around by
restricting it to the current filesystem. When the **real** cause turned out to be the `spack()`
framing defect above (which was actually captured in packet logs, and explains the intermittency),
`7f77234` was **reverted** (`6d3f1a4`, `BUGFIX_20260709_3.md`).

**Net effect on `ckufio.c`:** the `SPACMD` / `#ifdef BSD4` block is **byte-identical to
`base-20260616`** (still the pristine upstream all-volumes `df` on macOS/BSD — the `BSD4` guard is
dead, but deliberately kept upstream-verbatim). The only `ckufio.c` net changes in this range are an
explanatory comment above `SPACMD`, two incidental `clang-format` comment rewraps, and the V-6 /
V-21 / V-22 security fixes. `BUGFIX_20260709.md` carries a "REVERTED" notice at its top.

## 7. Behavioral changes a user could notice

- **Reliability:** transfers / REMOTE commands whose response tail hit exactly 90/91 data bytes
  (block-check 3) used to stall ~4.5 min then fail; now they complete immediately (§3).
- **New command `SET TRANSFER PIPES-RECEIVE ON|OFF`** (default **OFF**) — inbound `!command` pipe
  execution on *receipt* is no longer auto-enabled by `SET TRANSFER PIPES ON` (which now governs the
  SEND direction only). `SHOW TRANSFER` gains a `Transfer Pipes-Receive:` line; a peer requesting
  pipe execution with both flags on now prints a warning (V-11).
- **File-transfer safety:** received / FTP-GET filenames with a `..` component or absolute path are
  refused / stripped instead of traversing (V-6, V-12).
- **`SET KEY`** and **`OUTPUT`/`LINEOUT`** inputs long enough to overflow after escape-doubling now
  print `?Key definition too long` / `?Output text too long` and abort instead of corrupting memory
  (V-13, V-14).
- **`DIAL-COMMAND`/`EDITOR`/`BROWSER /OPTIONS`** values: `%` sequences other than the first `%s` are
  now literal text, not `printf` conversions (V-20, V-27).
- **FTP PASV** always dials back the control-connection host, ignoring the IP in the `227` reply —
  a real (edge-case) behavior change for multi-homed/NAT'd servers that rely on the reported address
  (V-16).
- **Unset `LANG`** now shows `LANG=""` instead of `LANG="(null)"` (V-37).
- **No change:** no-arg `REMOTE SPACE` on macOS/BSD is back to the pristine all-volumes `df` (§4).

## 8. Residual risks / known-incomplete items

Deliberately left open, per the docs:

1. **Unfixed findings:** V-30 (active-mode FTP data-conn peer race), V-31 (inert overflow
   *diagnostic*, not unsafe), V-33 (`TELOPT_NAWS` short-subneg stale read; default-off `sstelnet`
   server mode only). See `VULNERABILITIES-20260707.md` / `PROPOSED-FIXES-20260708.md`.
2. **`lpcapu==0` + block-check-3 gap** (`BUGFIX_20260709_2.md` §4.1): when long packets aren't
   negotiated, `sdata()`'s flat clamp isn't block-check-aware — data can still reach the 95-byte
   DEL-LEN case. Unreachable between two instances of this fork; only against an old/limited peer.
   The dead `maxdata()` has the correct bctl-aware formula if ever worth closing.
3. **`ttinl()` timer hygiene** (`BUGFIX_20260709_2.md` §4.2): the garbage-LEN early returns skip
   `ttimoff()`, leaving the SIGALRM timer armed — latent, masked by re-arming; left untouched
   because fixing the sender removes the only known trigger.
4. **`syscmd()` unbounded wait** (`BUGFIX_20260709_3.md` §4.1): every `syscmd()`-based server command
   (`REMOTE HOST/WHO/DIRECTORY/TYPE/PRINT`, `SPACE <dir>`) still waits indefinitely with a silent
   1 Hz empty-D-packet keepalive if the underlying shell command blocks — not specific to `SPACMD`.
   A real fix would need a cumulative no-output cap in the shared `sdata()`/`zinfill()` loop.
5. **Dead `#ifdef BSD4` branches** (`SPACMD`, sibling `WHOCMD`) are now **deliberately preserved
   upstream-verbatim** — do not "fix" them to `BSD44` without new evidence the all-volumes `df` is a
   real problem (`BUGFIX_20260709_3.md` §4.3).
6. **V-11 adjacent surface, V-22 end-to-end proof, V-27 placement edge case** — the three residuals
   in `FIXES-20260708.md` §"Known residuals".
